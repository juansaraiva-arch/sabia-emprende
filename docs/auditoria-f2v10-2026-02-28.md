## 1. RESUMEN EJECUTIVO

### Que tenemos
MDF PTY cuenta con una base solida para construir el borrador automatico del F2 V10:

- **Motor fiscal DGI ya existente** (`dgi-mappings.ts` + `EspejoDGI.tsx`): Mapea cuentas contables a los formularios 430 (ITBMS) y 03 (Renta). Ya pre-llena casillas y genera calendarios DGI.
- **Base de datos completa** (11 tablas Supabase): Sociedades con `entity_type` (SA/SRL/SE), `fiscal_regime` (se_exempt), `incorporation_date`, `tax_id` (RUC). Registros financieros mensuales con revenue, gastos desglosados, tax_expense.
- **Contabilidad formal**: Plan de cuentas Panama, libro diario con partidas dobles, libro mayor, balance de comprobacion.
- **Nomina (PayrollEngine)**: Calculo completo de CSS patronal, ISR empleado, XIII Mes, con persistencia localStorage.
- **Alertas de compliance**: Sistema de alertas que ya cubre vencimiento de renta anual (31 marzo), ITBMS trimestral, renta estimada trimestral.
- **Formalizacion S.E.**: Ruta completa de 6 pasos incluyendo AMPYME, RUC, Aviso Operacion.

### Que falta (brechas criticas)
1. **No existe tabla de facturas electronicas** — No hay un registro estructurado de facturas emitidas/recibidas. Solo referencias textuales en `journal_entries.reference`.
2. **No existe tabla de declaraciones** — No se guarda historial de borradores ni estado de presentacion en DB (solo un flag localStorage `midf_dgi_filed`).
3. **No existe datos de socios/directivos** — No hay tabla de Junta Directiva, accionistas ni porcentajes de participacion.
4. **No existe campo de fecha de constitucion en localStorage/frontend** — `incorporation_date` existe en DB pero la app funciona mayormente en modo demo sin backend.
5. **FECI no modelado** — No hay referencia al Fondo Especial de Compensacion de Intereses.
6. **F2 V10 no existe en el codebase** — Solo estan modelados Form 430 y Form 03. No hay ninguna referencia al formulario F2.

### Nivel de esfuerzo estimado
- **Esfuerzo total:** Alto (3-5 dias de desarrollo)
- **Riesgo principal:** Datos de facturacion electronica ausentes — el V10 valida contra SFEP
- **Recomendacion:** Implementar con datos disponibles, marcar FE como entrada manual, alertar sobre reconciliacion

## 5. RIESGOS IDENTIFICADOS

### RIESGO 1: Ausencia de datos de Facturacion Electronica (CRITICO)
- **Impacto:** El F2 V10 cruza ingresos declarados contra facturas emitidas en el SFEP. Sin datos de FE, no se puede reconciliar.
- **Mitigacion:** Crear campo de "Total FE declarado" como entrada manual. Alertar al usuario que la DGI validara contra su SFEP. En futuro: integracion con API de facturacion electronica.

### RIESGO 2: incorporation_date no disponible en modo demo (ALTO)
- **Impacto:** Sin fecha de constitucion, no se puede calcular exoneracion ISR. Error aqui = pagar impuesto que no debe, o no pagarlo cuando debe.
- **Mitigacion:** Solicitar fecha de constitucion como dato obligatorio al generar borrador. Guardar en localStorage si API no disponible.

### RIESGO 3: Usuarios sin datos financieros completos (MEDIO)
- **Impacto:** Borrador con campos vacios que no sirve para presentar.
- **Mitigacion:** Sistema de completitud (0-100%) con indicadores visuales claros. Nunca bloquear el borrador por datos faltantes.

### RIESGO 4: Datos de socios no modelados (MEDIO)
- **Impacto:** Seccion de distribucion de utilidades quedara vacia.
- **Mitigacion:** Formulario inline simple para agregar 1-3 socios con nombre, cedula, porcentaje.

### RIESGO 5: Confundir Form 03 generico con F2 V10 especifico (BAJO)
- **Impacto:** El Form 03 existente aplica a todas las personas juridicas. El F2 V10 es especifico para microemprendedores S.E.P.
- **Mitigacion:** Crear formulario F2 V10 separado que reutilice las casillas del Form 03 pero agregue logica S.E. (exoneracion, umbrales, tasa unica 0).
